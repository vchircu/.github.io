OpenID Connect in a nutshell

In the previous post we talked about delegated authorization with OAuth 2.0. After the success of OAuth 2.0, many companies tried using OAuth as anÂ authorization method, which has known pitfalls. The main issue is that if a malicious application manages to steal anÂ access token, it can use itÂ to impersonate the user. Many OAuth 2.0 providers modified the OAuth protocol to implement authorization, which made them less interoperable and possibly even vulnerable to impersonation attacks. Since OpenID was an existent standard for federated identity, there was interest in combining these twoÂ protocols, so the third generation of the OpenID protocol was built as an OAuth 2.0 extension.

OpenID Connect

OpenID Connect is an identity layer built on top of OAuth 2.0. ItÂ specifies a RESTfull HTTP API that is interoperable (it uses JSON as the data format), simple, mobile friendly and secure.

Since identity is an overloaded term, OpenID Connect definesÂ identityÂ  as aÂ set of claims (attributes)Â related to a an entityÂ (person, service, machine).

Actors

These are the actors that take part in the OpenID Connect authentication flow:

An OpenIDÂ IdentityÂ Provider (IDPÂ orÂ OP)Â is anÂ  OAuth 2.0 authorization server that offers authentication as a service.

AÂ Relying Party (RP)Â is anÂ OAuth 2.0 client that relies onÂ the IDP in order to authenticate users and request claims.

The End-User is the entity for which we request identity information. In OAuth 2.0 terms, this is theÂ resource owner, and the resource that he owns is his own identity.

ID Token

The OpenID Connect protocol specifies a new type of token – the ID Token – that encapsulates the user’s identity. The main attributes (claims) that a token contains are:

  • IssuerÂ  – the authorization server that issued thisÂ token
  • Subject – an identifier for the End-User
  • AudienceÂ – who can use this token; the client (Relying Pary) of the token must be in the audience of the token
  • ExpirationÂ  -the token’s expiration time

OpenID Connect mandates the use of JSON Web Token (JWT) as the format of the token. The token must be signed by the Authorization Server using JSON Web Signature (JWS),Â in order to achieve integrity and non repudiation. It mayÂ also be encrypted with JSON Web Encryption (JWE) for confidentiality.

Claims

OpenID Connect defines 20 standard claimsÂ (name, email, address, phone, etc.), but you can also defineÂ custom claims. The OpenID Connect specs also defines four standard scopes – profile, email, address and phone – which are groups ofÂ standard claims (e.g. the emailÂ scope contains the email and email_verified claims)Â .

The client application can request claims either in the initial authentication request or with an API call to the UserInfo endpoint.

Passport Control Analogy

OpenID can easily be explained by comparing it with the border control check that you have to go throughÂ in airports. When you (the End-User) go through passport control, you authenticate to the airport security personnel (the Relying Party) by showing them your Passport (ID token). The Passport is issued by a national authority (Identity Provider) that the airport employees trustÂ andÂ contains claims about you (name, age, nationality, etc.).

Registration

Authorization Flows

OpenID Connect defines three types ofÂ flows: Code, Implicit and Hybrid.

Authorization CodeÂ Flow

The OpenID Connect authorization code flowÂ is very similar to the OAuth 2.0 server side flow.Â The End-User gets redirected to the Authorization Server, which must authenticate the user and request consent for the client to accessÂ the user’s identity. After this step, the End-UserÂ is redirected to the client with anÂ authorization code as a query string parameter. The client can then exchange theÂ authorization code and the client’sÂ credentials for anÂ IDÂ token and an access tokenÂ throughÂ a server side (back channel) call to the token endpoint. Before using the ID token, the client must validate it.

 

ImplicitÂ Flow

The Implicit flowÂ is very similar to the OAuth 2.0 client side flow and it is best suited for client side applications. TheÂ ID token and, optionally, anÂ access token are returned from the authorization endpoint.Â Before using the ID token, the client must validate it.

HybridÂ Flow

The Hybrid flow is a combination of the Authorization Code and Implicit flows. It allowsÂ the backend of the client to receive anÂ authorization code, while the frontend can receive anÂ IDÂ tokenÂ and/orÂ anÂ access token.

Endpoints

An endpoint is simply an HTTP resource. OpenID ConnectÂ defines 4 coreÂ endpoints: 3Â authorization server endpoints â€“ authorization endpoint,Â token endpointÂ and UserInfo endpoint,Â and 1 client endpoint â€“ the redirect endpoint.

TheÂ Authorization EndpointÂ is used to ensure the user is authenticated and optionally grants access to other claims about his identity. If the user is not authenticated, then he will be redirected to the Identity Provider in order to authenticate.

The Token EndpointÂ is used to exchange the authorization grant for an IDÂ token and an access token.Â This endpoint may accept other OAuth 2.0 grant types in order to issue access tokens.

The UserInfo EndpointÂ is anÂ OAuth 2.0-protected resource.Â It can be used to request claims about the user, given that a valid access token is passed in the request.

The Redirect Endpoint is used to redirect the End-Userâ€™s browser back to the client application. Tokens areÂ also passed to the client through query string parameters or hash code URL fragments.

Apart fromÂ these 4 primaryÂ endpoints, the OpenID Connect specification also defines the following optional endpoints:

  • The Session Management Endpoint can be used by clients to check if a user has an active session or forÂ logging out.
  • The DiscoveryÂ EndpointÂ can beÂ used by the Relying Party in order to find out the OpenID Provider behind a given user (usually this can be infered fromÂ the user’s email address). Â OpenID Connect mandates the use of theÂ WebFingerÂ protocol for this use case.
  • The Identity Provider Metadata EndpointÂ can be used to request information about the OpenID Identity Provider. This metadata can contain endpoint URLs and supported capabilities (grant types, scopes, claims, signing and encryption algorithms, etc.)
  • The Dynamic Client Registration EndpointÂ can be used in order to dynamically registerÂ Relying Parties.

The last three endpoints enable a seamless integration and registration with an OpenID Provider – when an End-UserÂ tries to login for the first time usingÂ aÂ Relying Party,Â the clientÂ can get the URL of the IDP by using theÂ Discovery Endpoint, then use theÂ Identity Provider Metadata Endpoint to get the registration URLÂ of the provider and then automatically register using theÂ Dynamic Client Registration Endpoint.

JWT

JSON Web TokenÂ is a compact JSON format useful for exchanging informationÂ between parties. The OpenID Connect specification mandates the use of JWT as the format ofÂ the ID token.

The entire JWT is Base64Url encoded and is split in three parts, separated by dots:

  • Header – contains the type of the token – JWT – and the cryptographic operation applied on the body
  • Body –Â contains the claims set – the real business data. There are three types of claims:Â reservedÂ (a set of predefined claims like issuer, subject, audience, expiration, etc.),Â publicÂ (these can be defined by those using JWT, but must beÂ collision-resistant, either by beingÂ registered with IANA or by namespacing them with an URI) andÂ private (these are private claims, known onlyÂ by parties thatÂ agree on using them)
  • SignatureÂ – the Header, Body and a secret are signed using the cryptographic operation specifiedÂ in the header.

Conclusion

OpenID Connect is the emerging standard for federated identity. It’s an identity layer on top of OAuth 2.0, built usingÂ REST endpoints and JSON messages. It is more interoperable than previous solutions based on OAuth 2.0 and its adoption rate is growing more rapidlyÂ than for previous versions of OpenID.

Being a newer standardÂ than SAML, OpenID Connect has some advantages: it is simpler, more interoperable and mobile friendly, meaning OpenID Connect will gainÂ more traction in the enterprise space.
